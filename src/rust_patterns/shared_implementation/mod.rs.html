3# Information Hiding, Encapsulation, and Shared Implementation
4
5Rust supports shared interfaces via traits. However, Rust's support for
6shared _implementation_ is very limited.
7
8A trait can have a default implementation for a method, but this support for shared
9implementation is hamstrung by the fact that traits cannot have data members of any
10visibility. Thus, any default implementation can only assume about `self` what the trait
11itself asserts and cannot access any data directly. What's more, because they are intended
12as a mechanism for defining public interfaces, trait methods are necessarily public.
13
14Generics have similar limitations: they can assume very little about the type parameters,
15and they cannot access data members of any visibility. Their advantage is that they are
16monomorphized at compile time and thus don't incur the cost of runtime dynamic dispatch.
17
18For someone coming from OOP languages like C++, Rust's lack of language-level
19support for _shared implementation_ (not just shared interfaces) can be a pain
20point. This module demonstrates patterns for dealing with these limitations:
21
22- The sealed trait concept is an umbrella for a set of techniques for achieving fine-grained access and
23  overridability control of trait methods while also maintaining a level of encapsulation.
24- `impl`-side trait constraints and blanket trait implementations are patterns
25  for dealing with Rust's ophan/coherence rules and that can result in cleaner library interfaces.
26
27## Software Engineering 101.
28
29*Information hiding* means concealing the internal details (design decisions,
30data representations, or algorithms) of a software component so that other parts of the system
31depend only on its public interface, not on its inner workings.[¹](#info_hiding)
32
33| Term              | Definition                                                   | Relationship to Information Hiding                           |
34| ----------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
35| **Encapsulation** | The mechanism enforcing information hiding at the language level. | The implementation of information hiding in the source code. |
36| **Abstraction**   | The cognitive process of determining the essential characteristics of a thing without respect to its implementation. | The intellectual outcome of information hiding.              |
37| **Modularity**    | Having components ("modules") that are self-contained, have well-defined responsibilities and interfaces, and can be developed, tested, and maintained independently. | The system-level outcome of information hiding.              |
38
39One can (and *should*) quibble with these definitions, but they are good enough for our purpose.
40

42

44Parnas in his 1972 paper, *[“On the Criteria To Be Used in Decomposing Systems

46context, the word _module_ is synonymous with _software component_ and need
47not correspond directly to the Rust language feature of the same name.
