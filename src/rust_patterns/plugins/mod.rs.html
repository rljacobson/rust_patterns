3# Plugin Systems in Rust
4
5Writing a plugin system in Rust usually involves two main components:
6
71. A mechanism for providing a generic interface between the main program and the plugin.
82. A mechanism for plugin discovery, which you might call "initialization"
9  or "registration", but which is more about knowing which plugins exist,
10  possibly across crate boundaries, without explicit central declaration.
11
12The first is well-trodden ground in Rust. You typically use a trait to define
13the generic interface. (See [crate::type_erased_api].)
14
15For the second, the tricky part is avoiding having to know beforehand what plugins exist
16and what their dependencies are. The phrase "plugin" as we are using it here implies some
17degree of automatic discovery of these things. (Otherwise, they're just libraries or modules
18or, if you want to be maximally generic, *dependencies*.) We are especially interested
19the situation where you have a library that can use plugins defined in external crates.
20
21While some kind of automatic initialization is implied, note that the plugin problem itself is a
22separate problem from that of initialization (lazy or otherwise), which `OnceCell`, `LazyStatic`,
23and other mechanisms address. The issue is *not* that initialization needs to occur at all, but
24rather that we need to know which items exist in the first place. Once we know which items exist,
25our strategy for initialization, while interesting in its own right, is a separate question.
26
27There are two main ways to do plugins in Rust, plus hybrid approaches.
28
29## 1. “Run Before Main” Mechanisms
30
31These systems rely on functions that are automatically executed before
32`main()` is executed. They typically use compiler or linker attributes (`#[ctor]`,
33`#[dtor]`, or `#[used]`) to mark static constructors that run at startup.
34
35Example crates: [ctor](https://crates.io/crates/ctor), [init](https://crates.io/crates/init)
36
37This mechanism is easy to understand and enjoys wide platform
38support, because it's the same mechanism used by C++ for construction of
39statics, but it has severe limitations in Rust applications, including but not limited to:
40
41- unspecified execution order
42- [fragility in dynamic linking contexts](https://github.com/rust-lang/rust/issues/28794#issuecomment-368693049)
43  (cdylib, plugins, embedded targets).
44- [fragility in static linking contexts](https://github.com/mmastrac/rust-ctor#warnings) ("some linker configurations
45  may cause `#[ctor]` and `#[dtor]` functions to be stripped from the final binary")
46- execution of code prior to program initialization (e.g. using `println!` causes a panic)
47
48## 2. Linker-based "Distributed Slice" Mechanisms
49
50This category uses linker section merging to collect specially annotated static
51data items (not code) across crates into a single contiguous slice. (The static
52data could be function pointers or static structs with their own methods.)
53
54Example crates: [linkme](https://crates.io/crates/linkme),
55[distributed_slice](https://crates.io/crates/distributed_slice).
56
57This mechanism is true automatic cross-crate registration with *zero runtime overhead*, because
58it is literally just static data globally accessible by name. As with the `ctor` solution,
59order is not specified, which in practice just means that the question of initialization
60order is moved to user land at runtime, arguably where it belongs. Unfortunately, this
61mechanism (currently) relies on specific linker behavior and is not compatible with certain
62build environments and runtime targets. Crucially, *there is currently no support for Wasm.*
63
64## 3. Hybrid Approaches
65
66These approaches add layers of convenience, for example, ways to do lazy initialization and thread safety.
67
68- [inventory](https://crates.io/crates/inventory) — layers initialization on
69  top of the "distributed slice" concept for situations where initialization
70  requires runtime code execution, not just compile-time statics.
71- [static_init](https://crates.io/crates/static_init) — adds sophisticated initialization
72  mechanisms that claim to be superior to `lazy_static` and other alternatives for certain use cases.
73
74## A Not-Quite Plugin System
75
76We are requiring an "automatic discovery" mechanism for plugins in which we have access to every plugin (or plugin type)
77at runtime, even if it is defined in an external crate. But suppose we weaken this requirement to the following:
78
79> We want to be able to discover new plugins lazily as they are accessed.
80
81This is a very different proposition because it explicitly abandons the requirement that we
82know all the types up front. It's enough to discover the types as they are accessed, and
83if an item is never accessed at all, it isn't even initialized. The very act of accessing
84the type, presumably through a function with a generic parameter, tells us it exists.
85The [`crate::item_registry`] module demonstrates a system that can be used in this way.
86
87---
88
89The general consensus is that the "distributed slice" mechanism is the best
90way forward for the Rust ecosystem, but compiler support is ultimately
91required. The lack of Wasm support is a nonstarter for many applications.
92
93The reason the "distributed slice" mechanism is the "right" thing to do boils down to Rust's
94fundamental assumption that nothing executes before or after `main()`. Violating that runtime
95assumption will be fragile at best *in any conceivable future*. On the flip side, "run
96before main" has widespread platform support inherited from C++, so the "worse" solution is
97currently the best solution right now—indeed, the only solution if Wasm support is required.
98
99It's not ideal. Everybody knows this. It's just a hard nut to crack.
100
