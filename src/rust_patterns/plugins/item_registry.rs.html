3# The "Registry Pattern"
4
5The "Registry Pattern" is a plugin-like pattern for managing a collection of
6[`RegisteredItem`] instances such that discovery of items is automatic/implicit.
7
8For our use case, the [`RegisteredItem`]s are lazily registered (the allocation or
9instantiation of _static_ data) and instantiated (for _instance_ data), but this is not
10an essential part of the pattern. What _is_ essential is that the [`RegisteredItem`]s
11are automatically discovered, even if they are defined in external crates.
12
13In the simplest formulation, the "registry pattern" is just a vector of [`RegisteredItem`]s
14each one of which has a static `index` variable in which is stored its position in
15the vector that owns it. We just provide a mechanism for assigning the indices to
16the [`RegisteredItem`]s at runtime in a logically consistent and thread-safe way
17that doesn't require explicit knowledge of every [`RegisteredItem`] type that exists.
18
19We define the [`RegisteredItems`] struct as a wrapper type around the vector of [`RegisteredItem`]
20instances to provide convenient [`RegisteredItems::get_item`] and [`RegisteredItems::get_item_mut`]
21methods, which automatically instantiate the [`RegisteredItem`] _instance_ if it has
22not yet been initialized. Note that we intentionally do not call the [`RegisteredItems`]
23struct a "registry" because it is not necessarily a singleton. In fact, we might have many
24[`RegisteredItems`] instances each holding `RegisteredItem` instances it owns. "The registry"
25that records which `RegisteredItem` types exist is purely abstract; depending on the underlying
26implementation, there might not be any actual list of which `RegisteredItem` types exist at all!
27
28A [`RegisteredItem`] need not actually be a singleton, but it, or rather its _type_, is only ever
29_registered_ once at runtime; that is, its `index`, which is shared among all instances of the
30type, is initialized only once. This allows a "registered item" type to have "singleton metadata"
31[ยน](#note1) associated with it in addition to whatever instance data it needs. As already mentioned,
32at minimum each [`RegisteredItem`] knows its own `index`, which is used to access the item from
33the owning [`RegisteredItems`] container via simple indexing into a vector of [`RegisteredItem`]s.
34(See the ['plugins' module](crate::plugins) for more details about implementation.)
35
36
37## Q: Wait, aren't you just talking about static variables? Or, aren't you just talking about lazy initialization of static data?
38
39A: No, I am describing an interface for *automatic* discovery of special types
40and, (though only incidentally) the initialization of their static data. How rich
41the static data is depends entirely on your use case, so long as it at minimum has
42an `index` field to make accessing instances of the items in the owning container easy.
43
44## Notes
45<a name="note1">1.</a>
46This is usually just called _static_ data, or data in a _static variable_, but I am trying
47to avoid any assumptions about when this data is initialized and whether it is mutable.
48
49*/

63/// A trait for items that can be registered (`DataPlugin`, `PersonProperty`)
64pub trait RegisteredItem: Any {
65    /// Convenient for debugging.
66    fn name() -> &'static str
67    where
68        Self: Sized;
69
70    /// The index of the item in the [`RegisteredItem`] list, typically just a vector that lives in the parent.
71    fn index() -> usize
72    where
73        Self: Sized;
74
75    /// Creates a new instance of the item.
76    fn new_boxed() -> Box<Self>
77    where
78        Self: Sized;
79}

81/// A boxed [`RegisteredItem`]
82pub type BxRegisteredItem = Box<dyn RegisteredItem>;

84/// Global item index counter, keeps track of the index that will be assigned to the next item that
85/// requests an index. Equivalently, holds a *count* of the number of items currently registered.
86pub static NEXT_ITEM_INDEX: Mutex<usize> = Mutex::new(0);
87
88/**
89For simple applications, just knowing how many items are registered is
90enough, and that information is already captured in [`NEXT_ITEM_INDEX`]. In
91more sophisticated implementations, we would want to store metadata about the
92items that are registered. For example, we might want to track dependencies
93between registered items. We could store that metadata in a static variable,
94or we could have a global data store like the following.
95
96The downside of using a static variable to store the metadata is that client
97code would be responsible for its implementation. If instead we use a global
98data store, then the client code can just use the `add_to_registry` method to
99add items to the store and wouldn't need to worry about the implementation.
100*/
101#[allow(unused)]
102pub static REGISTERED_ITEMS: LazyLock<Mutex<RefCell<HashSet<TypeId>>>> =
103    LazyLock::new(|| Mutex::new(RefCell::new(HashSet::default())));
104
105/// Adds a new item to the registry. The job of this method is to create whatever "singleton"
106/// data/metadata is associated with the [`RegisteredItem`] if it doesn't ready exist. The "registry"
107/// is a global singleton (literally just [`NEXT_ITEM_INDEX`] in this implementation), so it is
108/// enough to know the concrete type of the [`RegisteredItem`]. For this illustration, the only singleton
109/// metadata that is instantiated is the item's index, which we return in case the caller wants it.
110///
111/// We could also interpret this method as a lazy getter for the static metadata associated with the
112/// item.
113pub fn add_to_registry<R: RegisteredItem>() -> usize {
114    R::index()
115    // If we had a global store for the metadata, we would do something like this:
116    // REGISTERED_ITEMS
117    //     .lock()
118    //     .unwrap()
119    //     .borrow_mut()
120    //     .insert(<R as RegisteredItem>::type_id());
121}
122
123/// An accessor for `NEXT_ITEM_INDEX`
124pub fn get_data_plugin_count() -> usize {
125    *NEXT_ITEM_INDEX.lock().unwrap()
126}
127
128/// Acquires a global lock on the next available plugin index, but only increments
129/// it if we successfully initialize the provided index. The `index` of a registered
130/// item is assigned at runtime but only once per type. It's possible for a single
131/// type to attempt to initialize its index multiple times from different threads,
132/// which is why all this synchronization is required. However, the overhead
133/// is negligible, as this initialization only happens once upon first access.
134pub fn initialize_item_index(plugin_index: &AtomicUsize) -> usize {
135    // Acquire a global lock.
136    let mut guard = NEXT_ITEM_INDEX.lock().unwrap();
137    let candidate = *guard;
138
139    // Try to claim the candidate index. Here we guard against the potential race condition that
140    // another instance of this plugin in another thread just initialized the index prior to us
141    // obtaining the lock. If the index has been initialized beneath us, we do not update
142    // [`NEXT_ITEM_INDEX`], we just return the value `plugin_index` was initialized to.
143    // For a justification of the data ordering, see:
144    //     https://github.com/CDCgov/ixa/pull/477#discussion_r2244302872
145    match plugin_index.compare_exchange(usize::MAX, candidate, Ordering::AcqRel, Ordering::Acquire)
146    {
147        Ok(_) => {
148            // We won the race โ increment the global next plugin index and return the new index
149            *guard += 1;
150            candidate
151        }
152        Err(existing) => {
153            // Another thread beat us โ donโt increment the global next plugin index,
154            // just return existing
155            existing
156        }
157    }
158}

160pub struct RegisteredItems {
161    /// In this pattern, the [`RegisteredItem`]s are stored in a `Vec<Box<dyn Any>>`.
162    items: Vec<OnceCell<Box<dyn Any>>>,
163
164    _phantom: std::marker::PhantomData<dyn RegisteredItem>,
165}
166
167impl RegisteredItems {
168    /// Fetches an immutable reference to the item `R` from the registry.
169    #[must_use]
170    pub fn get_item<R: RegisteredItem>(&self) -> &R {
171        let index = R::index();
172        self.items
173        .get(index)
174        .unwrap_or_else(|| panic!("No registered item found with index = {index:?}. You must use the `define_registered_item!` macro to create a registered item."))
175        .get_or_init(|| R::new_boxed())
176        .downcast_ref::<R>()
177        .expect("TypeID does not match registered item type. You must use the `define_registered_item!` macro to create a registered item.")
178    }
179
180    /// Fetches a mutable reference to the item `R` from the registry.
181    #[must_use]
182    pub fn get_item_mut<R: RegisteredItem>(&mut self) -> &mut R {
183        let mut self_shadow = self;
184        let index = R::index();
185
186        // If the item is already initialized, return a mutable reference.
187        // Use polonius to address borrow checker limitations.
188        polonius!(|self_shadow| -> &'polonius mut R {
189            if let Some(any) = self_shadow.items[index].get_mut() {
190                polonius_return!(
191                    any.downcast_mut::<R>()
192                        .expect("TypeID does not match registered item type")
193                );
194            }
195            // Else, don't return. Fall through and initialize.
196        });
197
198        // Initialize the item.
199        let cell = self_shadow
200        .items
201        .get_mut(index)
202        .unwrap_or_else(|| panic!("No registered item found with index = {index:?}. You must use the `define_registered_item!` macro to create a registered item."));
203        let _ = cell.set(R::new_boxed());
204        cell.get_mut()
205        .unwrap()
206        .downcast_mut::<R>()
207        .expect("TypeID does not match registered item type. You must use the `define_registered_item!` macro to create a registered item.")
208    }
209}

211/// This macro ensures correct implementation of the `RegisteredItem` trait. The tricky bit is the implementation of
212/// `RegisteredItem::index`.
213#[macro_export]
214macro_rules! registered_item_impl {
215    ($item_name:ident) => {
216        impl $crate::item_registry::RegisteredItem for $item_name {
217            fn name() -> &'static str
218            where
219                Self: Sized,
220            {
221                stringify!($item_name)
222            }
223
224            fn index() -> usize {
225                // This static must be initialized with a compile-time constant expression.
226                // We use `usize::MAX` as a sentinel to mean "uninitialized". This
227                // static variable is shared among all instances of this concrete item type.
228                static INDEX: std::sync::atomic::AtomicUsize =
229                    std::sync::atomic::AtomicUsize::new(usize::MAX);
230
231                // Fast path: already initialized.
232                let index = INDEX.load(std::sync::atomic::Ordering::Relaxed);
233                if index != usize::MAX {
234                    return index;
235                }
236
237                // Slow path: initialize it.
238                $crate::item_registry::initialize_data_plugin_index(&INDEX)
239            }
240
241            fn new() -> Box<Self> {
242                Box::new($item_name)
243            }
244        }
245
246        $crate::paste::paste! {
247            $crate::ctor::declarative::ctor!{
248                #[ctor]
249                fn [<_register_item_$item_name:snake>]() {
250                    $crate::item_registry::add_to_registry::<$item_name>()
251                }
252            }
253        }
254    };
255}
