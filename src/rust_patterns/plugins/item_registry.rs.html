2# The Registry Pattern
3
4The *registry pattern* is a plugin-like pattern for managing a collection of
5[`RegisteredItem`] instances such that discovery of items is automatic/implicit.
6
7For our use case, the [`RegisteredItem`]s are lazily registered (the allocation or
8instantiation of _static_ data) and instantiated (for _instance_ data), but this is not
9an essential part of the pattern. What _is_ essential is that the [`RegisteredItem`]s
10are automatically discovered, *even if they are defined in external crates*.
11
12> **Note:** If you are happy with the _lazy_ discovery of [`RegisteredItem`]s, then you
13> can get away with a simpler implementation that uses neither "run before `main()`" nor
14> "distributed slice" mechanisms (see ['plugins' module](crate::plugins) for discussion).
15> There is a trade-off: You need a mechanism for interior mutability of `RegisteredItems`
16> (the owner of the item instances) if you want to initialize the [`RegisteredItem`] on
17> first access, because you don't know how many [`RegisteredItem`] types there are globally
18> when [`RegisteredItems`] is constructed and thus don't know how big the vector should be.
19
20In the simplest formulation, the "registry pattern" is just a vector of [`RegisteredItem`]s, one instance for each concrete [`RegisteredItem`] type that exists,
21each one of which having a static `index` variable in which is stored its position in
22the vector that owns it. We just provide a mechanism for assigning the indices to
23the concrete [`RegisteredItem`] types at runtime in a logically consistent and thread-safe way
24that doesn't require explicit knowledge of every [`RegisteredItem`] type that exists. To recap: We have one item instance per concrete [`RegisteredItem`]  type per store.
25
26In the next section we go into much more detail about the design decisions one has to make for any given implementation of the registry pattern. In that section we will point out which decisions we've made for the example we are illustrating in the code provided here, but here's a brief rundown of the main points: We define the [`RegisteredItems`] struct as a wrapper type around the vector of [`RegisteredItem`]
27instances to provide convenient [`RegisteredItems::get_item`] and [`RegisteredItems::get_item_mut`]
28methods, which automatically instantiate the [`RegisteredItem`] _instance_ if it has
29not yet been initialized. Note that we intentionally do not call the [`RegisteredItems`]
30struct a "registry" because it is not necessarily a singleton. In fact, we might have many
31[`RegisteredItems`] instances each holding `RegisteredItem` instances it owns. "The registry"
32that records which `RegisteredItem` types exist is purely abstract; depending on the underlying
33implementation, there might not be any actual list of which `RegisteredItem` types exist at all!
34
35A [`RegisteredItem`] need not actually be a singleton, but it, or rather its _type_, is only ever
36_registered_ once at runtime; that is, its `index`, which is shared among all instances of the
37type, is initialized only once (in a `ctor`). This allows a "registered item" type to have "singleton metadata"
38[ยน](#note1) associated with it in addition to whatever instance data it needs. As already mentioned,
39at minimum each [`RegisteredItem`] knows its own `index`, which is used to access the item from
40the owning [`RegisteredItems`] container via simple indexing into a vector of [`RegisteredItem`]s.
41(See the ['plugins' module](crate::plugins) for more details about implementation.)
42
43
44## Q: Wait, aren't you just talking about static variables? Or, aren't you just talking about lazy initialization of static data?
45
46A: No, I am describing an interface for *automatic* discovery of special types
47and, (though only incidentally) the initialization of their static data. How rich
48the static data is depends entirely on your use case, so long as it at minimum has
49an `index` field to make accessing instances of the items in the owning container easy.
50

55to avoid any assumptions about when this data is initialized and whether it is mutable.

57# Design Options for the Registry Pattern
58
59In this section we explore in detail the different design choices one must make when implementing the registry pattern. We will call the `RegisteredItem` types of a *particular* implementation of the registry pattern the `Widget` types, or similar made up name. The owner will be some type wrapping a vector, which we'll call the `WidgetStore`, or some variant of `*Store`.
60
61## Design Dimensions for `Widget`s and `WidgetStore`s
62
63Below is my attempt at carving up the design space for our registry pattern implementation with a few alternatives listed for each. I'm using the phrase *dimension* to refer to the major orthogonal design choices that need to be made about our container. By *orthogonal* I mean a choice in one dimension does not depend on a choice in another dimension. In actual practice, implementation choices in one dimension may affect implementation in another dimension, so this independence is approximate and "leaky".
64
65All registered items that participate in the same registry pattern implementation must share the semantics implied by these design choices to avoid significant complexity explosion.
66
67**Initialization time**
68
69- In a singleton at program start (a la "distributed slice", prototype style)
70- Upon `WidgetStore` creation (an instance for each `Widget` type that exists is eagerly created)
71- Lazily, e.g. upon first access in `WidgetStore`. The "slots" in the owning `WidgetStore` are occupied by `OnceCell`s
72- Something else? E.g. super lazily: upon write access in `WidgetStore`.
73
74*In this example:* We lazily instantiate `Widget`s upon first access. We use `ctor` magic to know how many "slots" the `WidgetStore` needs to allocate.
75
76**Ownership semantics:** Specific to `WidgetStore`'s relationship to `Widget` values, _not_ to views client code have into the data.
77
78- `Widget`s are owned by the `WidgetStore` which gives out regular mutable or immutable references
79- A `WidgetStore` is either literally or essentially a singleton (possibly a naked "distributed slice"), only one instance of each `Widget` type is created globally.
80- `Widget`s are shared, stored in `Arc`/`Rc` or `Cow` types.
81- `Widget`s are `Copy` types
82- Move semantics. I'm not sure what the use case would be, but it's possible to move `Widget` instances out of the `WidgetStore`.
83
84*In this example:* `Widget`s are owned by the `WidgetStore` which gives out regular mutable or immutable references
85
86**Interior mutability**
87
88- "default" semantics: `Widget`s are owned by the `WidgetStore`, which is in turn owned by the `Context` (say), and they are all mutable or immutable at once accordingly.
89- `WidgetStore` lives in a `RefCell`, so entire store is mutable at once or immutable at once
90- `WidgetStore` holds individual `Widget`s in `RefCell`s, so multiple distinct `Widget`s can be simultaneously mutated
91- `Widget`s are `Copy` types, so they can be copied around and mutated in place (possibly as atomic types)
92- `Widget`s implement their own interior mutability
93
94*In this example:* "default" semantics (modulo `OnceCell`'s internal mechanisms); Mutation requires mutable access to the `WidgetStore`
95
96**Access control:** what "views" into the data are enabled
97
98- Client code get's "direct" access to a `&Widget` or `&mut Widget`
99- Client code passes a closure to a `with_widget` method, which closure is passed a `&Widget`; and similarly for mutable variant
100- `WidgetStore` provides rich API for interacting with `Widget`s, e.g. `WidgetStore::froculate_widget::<Sprocket>()` froculate's a widget on the caller's behalf. (We generally do this with `DataPlugin`s, for example.)
101- Pass out `Ref<'a, Widget>`s / `RefMut<'a, Widget>`s  to client code, which implement `Deref`/`DerefMut` to `&Widget`/`&mut Widget`
102- `Widget`s have `Copy` semantics, `WidgetStore::get()` returns a copy of the stored `Widget`. Have a corresponding `WidgetStore::set()` instead of `WidgetStore::get_mut()` with reference semantics.
103- YOLO raw pointers ๐คฃ
104
105*In this example:* Client code gets "direct" access to a `&Widget` or `&mut Widget`.
106
107**Other Dimensions:** Some other dimensions I don't have much to say about.
108
109- **Type Discovery:** We want "automatic" discovery, but some designs can get away with not using `ctor`/`linkme` tricks, in particular when discovery can happen on an as-needed (we "discover" `ConcreteWidget` only when `WidgetStore::get::<ConcreteWidget>()` is called); see notes below. This example uses `ctor` to know how many concrete `Widget` types there are globally and to assign each an index.
110- **Concurrency Model:** In this example, we take care to initialize each `Widget` type's index in a thread-safe way but otherwise don't share access across thread boundaries.
111- **Lifetime Model:** How long are `Widget` and `WidgetStore` instances expected to live? This implementation makes no assumptions.
112
113**"Leaky" Implementation Bits:**
114
115- `OnceCell` has its own interior mutability mechanism; same with `inventory`, others; though they can be optimized for specific use cases (e.g. `OnceCell` is fast to access after initialization)
116- If `WidgetStore` does not need to be immutable, or if it or its wrapped vector can live in a `RefCell`, then magic `ctor`/`linkme` mechanisms are not necessary, because we don't have to know the total count of all `Widget` types upon `WidgetStore` creation--we can grow the vector as needed.
117- If `WidgetStore` needs to be immutable, then we need to know the total count of all `Widget` types upon `WidgetStore` creation in order to allocate enough "slots" for all the `Widget` type _even if we defer `Widget` creation by using `OnceCell`s_.
118  - `OnceCell`s push (a fast form of) interior mutability out from the `Context` or `WidgetStore` level to the individual `Widget` level
119  - Eager initialization is an alternative to `OnceCell` and isn't necessarily harder to implement, since you already pay the price of fancy linker tricks for the `OnceCell` solution.
120
121## Multiple registries for different _thing_ categories vs. a single registry for all (Generic) Registered Items
122
123We can imagine two different approaches to how many implementations of the registry pattern are in a given project:
124
1251. A single implementation of the registry pattern could conceivably be generic over every category of *thing* that can be registered, or at least every category that matches all of your choices for you have made in all of the dimensions listed in the previous section.
126   - Unifies the storage and access for `Widget`s, `Gadget`s, `Blogets`, `Sprockets`, ..., potentially lots of disparate categories of objects.
127
128   - Minimizes lines of code, API
1292. You have separate implementations in your project of the registry pattern for each category even if the design choices in all dimensions of the previous section are the same. For example, you might implement the registry pattern for your `Widget` system and then have a separate independent implementation of the registry pattern for your `Gadget` system.
130   - Different public API for different categories of objects
131   - If we want to change the storage / ownership / access semantics for a category of objects, it's already distinguished from other categories and has its own API.
132
133My advice is to choose (2) almost always. Superficial implementation features are generally poor decomposition criteria. A `Widget` system and a `Gadget` system should generally not be artificially coupled.
134*/

148/// A trait for items that can be registered (`DataPlugin`, `PersonProperty`)
149pub trait RegisteredItem: Any + Default {
150    /// Convenient for debugging.
151    fn name() -> &'static str
152    where
153        Self: Sized;
154
155    /// The index of the item in the [`RegisteredItem`] list, typically just a vector that lives in the parent.
156    fn index() -> usize
157    where
158        Self: Sized;
159
160    /// Creates a new instance of the item.
161    fn new_boxed() -> Box<Self>
162    where
163        Self: Sized {
164        Box::new(Default::default())
165    }
166}
167
168/// A boxed [`RegisteredItem`]
169pub type BxRegisteredItem = Box<dyn RegisteredItem>;
170
171/// Global item index counter, keeps track of the index that will be assigned to the next item that
172/// requests an index. Equivalently, holds a *count* of the number of items currently registered.
173///
174/// If the "registry pattern" is implemented using "run before `main()`" (which is what we do in
175/// this example code; see ['plugins' module](crate::plugins) for discussion), `NEXT_ITEM_INDEX`
176/// will hold the total count of all registered items by the time any client code runs,
177/// because each type implementing the `RegisteredItem` trait calls `add_to_registry()` before
178/// the program's `main()` method is called. If the implementation uses the "distributed
179/// slice" mechanism instead, then `NEXT_ITEM_INDEX` increases monotonically as each type
180/// implementing the `RegisteredItem` initializes its own `index` static variable, which
181/// occurs according to whatever initialization strategy you adopt for `RegisteredItem` types.
182pub static NEXT_ITEM_INDEX: Mutex<usize> = Mutex::new(0);
183
184/// For simple applications, just knowing how many items are registered is
185/// enough, and that information is already captured in [`NEXT_ITEM_INDEX`]. In
186/// more sophisticated implementations, we would want to store metadata about the
187/// items that are registered.
188///
189/// For example, we might want to track dependencies between registered items. We could
190/// store that metadata in a static variable, or we could have a global data store like
191/// the following. The downside of using a static variable to store the metadata is
192/// that client code would be responsible for its implementation. If instead we use a
193/// global data store, then the client code can just use the `add_to_registry` method
194/// to add items to the store and wouldn't need to worry about the implementation.
195#[allow(unused)]
196pub static REGISTERED_ITEMS: LazyLock<Mutex<RefCell<HashSet<TypeId>>>> =
197    LazyLock::new(|| Mutex::new(RefCell::new(HashSet::default())));
198
199/// Adds a new item to the registry. The job of this method is to create whatever "singleton"
200/// data/metadata is associated with the [`RegisteredItem`] if it doesn't already exist. The "registry"
201/// is a global singleton (literally just [`NEXT_ITEM_INDEX`] in this implementation), so it is
202/// enough to know the concrete type of the [`RegisteredItem`]. For this illustration, the only singleton
203/// metadata that is instantiated is the item's index, which we return in case the caller wants it.
204///
205/// We could also interpret this method as a lazy getter for the static metadata associated with the
206/// item.
207pub fn add_to_registry<R: RegisteredItem>() -> usize {
208    R::index()
209    // If we had a global store for the metadata, we would do something like this:
210    // REGISTERED_ITEMS
211    //     .lock()
212    //     .unwrap()
213    //     .borrow_mut()
214    //     .insert(<R as RegisteredItem>::type_id());
215}
216
217/// An accessor for `NEXT_ITEM_INDEX`
218pub fn get_registered_item_count() -> usize {
219    *NEXT_ITEM_INDEX.lock().unwrap()
220}
221
222/// Encapsulates the synchronization logic for initializing an item's index.
223///
224/// Acquires a global lock on the next available item index, but only increments
225/// it if we successfully initialize the provided index. The `index` of a registered
226/// item is assigned at runtime but only once per type. It's possible for a single
227/// type to attempt to initialize its index multiple times from different threads,
228/// which is why all this synchronization is required. However, the overhead
229/// is negligible, as this initialization only happens once upon first access.
230///
231/// In fact, for the example implement provided here, we know we are calling
232/// this function once for each type in each `RegisteredItem`'s `ctor` function,
233/// which should be the only time this method is ever called for the type.
234pub fn initialize_item_index(plugin_index: &AtomicUsize) -> usize {
235    // Acquire a global lock.
236    let mut guard = NEXT_ITEM_INDEX.lock().unwrap();
237    let candidate = *guard;
238
239    // Try to claim the candidate index. Here we guard against the potential race condition that
240    // another instance of this plugin in another thread just initialized the index prior to us
241    // obtaining the lock. If the index has been initialized beneath us, we do not update
242    // [`NEXT_ITEM_INDEX`], we just return the value `plugin_index` was initialized to.
243    match plugin_index.compare_exchange(usize::MAX, candidate, Ordering::AcqRel, Ordering::Acquire)
244    {
245        Ok(_) => {
246            // We won the race โ increment the global next plugin index and return the new index
247            *guard += 1;
248            candidate
249        }
250        Err(existing) => {
251            // Another thread beat us โ donโt increment the global next plugin index,
252            // just return existing
253            existing
254        }
255    }
256    // An argument that the ordering given in the call to
257    // `compare_exhange` above is minimally constrained follows.
258    //
259    // On success (`plugin_index == usize::MAX`), we write the new value (candidate) to the atomic.
260    //
261    // **The Release ordering (write side):** Prevents previous writes from
262    // being reordered after the atomic store. Guarantees that any thread which
263    // later acquires this atomic will see all the writes that happened-before.
264    //
265    // **The Acquire ordering (read side):** Prevents subsequent reads from being moved
266    // before the successful `compare_exchange`. Ensures we see all memory effects that
267    // happened-before any prior release (of which there are none in our case). On failure,
268    // we only read the existing value from the atomic, so we want to guarantee we do not
269    // read a stale value. So I think we only need the `Ordering::Acquire` ordering, which
270    // prevents the read from being moved before the previous successful `compare_exchange`.
271    //
272    // We only need `Ordering::AcqRel` instead of `Ordering::SeqCst` for the success case,
273    // because we only care about the relative ordering of the reads/writes across threads
274    // rather than a single total global ordering all threads agree on. ([The difference is
275    // subtle](https://en.cppreference.com/w/cpp/atomic/memory_order.html#Sequentially-consistent_ordering),
276    // to put it mildly.)
277}

280/// A wrapper around a vector of [`RegisteredItem`]s.
281pub struct RegisteredItems {
282    items: Vec<OnceCell<Box<dyn Any>>>,
283}
284
285impl RegisteredItems {
286
287    /// Creates a new [`RegisteredItems`] instance, allocating the exact number
288    /// of slots as there are types that implement [`RegisteredItem`]s.
289    ///
290    /// This method assumes all types implementing [`RegisteredItem`] have been implemented
291    /// _correctly_. This is one of the pitfalls of this pattern: there is no guarantee
292    /// that types implementing [`RegisteredItem`] implemented a `ctor` that correctly
293    /// initializes its `index` and so forth. We can have at least some confidence,
294    /// though, in their correctness by supplying a correct implementation via a macro.
295    ///
296    /// Observe that we create an empty `OnceCell` in each slot in this implementation, but
297    /// we could just as easily eagerly initialize the [`RegisteredItem`] instances here
298    /// instead (possibly by iterating over constructor functions from [`REGISTERED_ITEMS`]).
299    pub fn new() -> Self {
300        let num_items = get_registered_item_count();
301        Self {
302            items: (0..num_items).map(|_| OnceCell::new()).collect(),
303        }
304    }
305
306    /// Fetches an immutable reference to the item `R` from the registry. This
307    /// implementation lazily instantiates the item if it has not yet been instantiated.
308    #[must_use]
309    pub fn get<R: RegisteredItem>(&self) -> &R {
310        let index = R::index();
311        self.items
312        .get(index)
313        .unwrap_or_else(|| panic!("No registered item found with index = {index:?}. You must use the `define_registered_item!` macro to create a registered item."))
314        .get_or_init(|| R::new_boxed())
315        .downcast_ref::<R>()
316        .expect("TypeID does not match registered item type. You must use the `define_registered_item!` macro to create a registered item.")
317    }
318
319    /// Fetches a mutable reference to the item `R` from the registry. This
320    /// implementation lazily instantiates the item if it has not yet been instantiated.
321    #[must_use]
322    pub fn get_mut<R: RegisteredItem>(&mut self) -> &mut R {
323        let mut self_shadow = self;
324        let index = R::index();
325
326        // If the item is already initialized, return a mutable reference.
327        // Use polonius to address borrow checker limitations.
328        polonius!(|self_shadow| -> &'polonius mut R {
329            if let Some(any) = self_shadow.items[index].get_mut() {
330                polonius_return!(
331                    any.downcast_mut::<R>()
332                        .expect("TypeID does not match registered item type")
333                );
334            }
335            // Else, don't return. Fall through and initialize.
336        });
337
338        // Initialize the item.
339        let cell = self_shadow
340        .items
341        .get_mut(index)
342        .unwrap_or_else(|| panic!("No registered item found with index = {index:?}. You must use the `define_registered_item!` macro to create a registered item."));
343        let _ = cell.set(R::new_boxed());
344        cell.get_mut()
345        .unwrap()
346        .downcast_mut::<R>()
347        .expect("TypeID does not match registered item type. You must use the `define_registered_item!` macro to create a registered item.")
348    }
349}

351/// This macro ensures correct implementation of the `RegisteredItem` trait. The tricky bit is the implementation of
352/// `RegisteredItem::index`, which requires synchronization in multithreaded runtimes. This is an instance of
353/// _correctness via macro_.
354#[macro_export]
355macro_rules! registered_item_impl {
356    ($item_name:ident) => {
357        impl $crate::plugins::item_registry::RegisteredItem for $item_name {
358            fn name() -> &'static str
359            where
360                Self: Sized,
361            {
362                stringify!($item_name)
363            }
364
365            fn index() -> usize {
366                // This static must be initialized with a compile-time constant expression.
367                // We use `usize::MAX` as a sentinel to mean "uninitialized". This
368                // static variable is shared among all instances of this concrete item type.
369                static INDEX: std::sync::atomic::AtomicUsize =
370                    std::sync::atomic::AtomicUsize::new(usize::MAX);
371
372                // Fast path: already initialized.
373                let index = INDEX.load(std::sync::atomic::Ordering::Relaxed);
374                if index != usize::MAX {
375                    return index;
376                }
377
378                // Slow path: initialize it.
379                $crate::plugins::item_registry::initialize_item_index(&INDEX)
380            }
381        }
382
383        $crate::paste::paste! {
384            $crate::ctor::declarative::ctor!{
385                #[ctor]
386                fn [<_register_item_$item_name:snake>]() {
387                    $crate::plugins::item_registry::add_to_registry::<$item_name>();
388                }
389            }
390        }
391    };
392}

442    // Test the internal synchronization mechanisms of `initialize_item_index()`.
443    //
444    // It is convenient to only have a single test that mutates `NEXT_ENTITY_INDEX`,
445    // because we can assume no other thread is incrementing it and can therefore
446    // test the value of `NEXT_ENTITY_INDEX` at the beginning and then at the end of
447    // the test.
448    //
449    // Note that this doesn't really interfere with other tests involving `RegisteredItems`,
450    // because at worst `RegisteredItems` will just allocate addition slots for
451    // nonexistent items, which will never be requested with a `get()` call.
452    #[test]
453    fn test_initialize_item_index_concurrent() {
454        // Test 1: Try to initialize a single index from multiple threads simultaneously.
455        let initial_registered_items_count = get_registered_item_count();
456
457        const NUM_THREADS: usize = 100;
458        let index = Arc::new(AtomicUsize::new(usize::MAX));
459        let barrier = Arc::new(Barrier::new(NUM_THREADS));
460
461        let handles: Vec<_> = (0..NUM_THREADS)
462            .map(|_| {
463                let index_clone = Arc::clone(&index);
464                let barrier_clone = Arc::clone(&barrier);
465
466                thread::spawn(move || {
467                    // Wait for all threads to be ready
468                    barrier_clone.wait();
469                    // All threads try to initialize at once
470                    initialize_item_index(&index_clone)
471                })
472            })
473            .collect();
474
475        let results: Vec<usize> = handles.into_iter().map(|h| h.join().unwrap()).collect();
476
477        let first = results[0];
478
479        // The index should be initialized
480        assert_ne!(first, usize::MAX);
481        // All threads should get the same index
482        assert!(results.iter().all(|&r| r == first));
483        // And that index should be what was originally the next available index
484        assert_eq!(first, initial_registered_items_count);
485
486
487        // Test 2: Try to initialize multiple indices from multiple threads simultaneously.
488        //
489        // Creates 5 different entities (each with their own atomic). Initializes
490        // each from a separate thread. Verifies they receive sequential,
491        // unique indices. Confirms the global counter matches the entity count.
492
493        // W
494        let initial_registered_items_count = get_registered_item_count();
495
496        // Create multiple different entities (each with their own atomic)
497        const NUM_ENTITIES: usize = 5;
498        let entities: Vec<_> = (0..NUM_ENTITIES)
499            .map(|_| Arc::new(AtomicUsize::new(usize::MAX)))
500            .collect();
501
502        let mut handles = vec![];
503
504        // Initialize each entity from a different thread
505        for entity in entities.iter() {
506            let entity_clone = Arc::clone(entity);
507            let handle = thread::spawn(move || {
508                initialize_item_index(&entity_clone)
509            });
510            handles.push(handle);
511        }
512
513        // Collect results
514        let mut results = vec![];
515        for handle in handles {
516            results.push(handle.join().unwrap());
517        }
518
519        // Each entity should get a unique, sequential index starting with `initial_registered_items_count`.
520        results.sort();
521        for (i, &result) in results.iter().enumerate() {
522            assert_eq!(result, i + initial_registered_items_count, "Entity should have index {}, got {}", i, result);
523        }
524
525        // Test 3: Try to initialize multiple entities from multiple threads multiple times.
526
527        // We account for the fact that some entities have been initialized
528        // in their `ctors`, so the indices we create don't start with 0.
529        let initial_registered_items_count = get_registered_item_count();
530
531        // Create 3 entities
532        let entity1 = Arc::new(AtomicUsize::new(usize::MAX));
533        let entity2 = Arc::new(AtomicUsize::new(usize::MAX));
534        let entity3 = Arc::new(AtomicUsize::new(usize::MAX));
535
536        let mut handles = vec![];
537
538        // Multiple threads racing on each of entity1, entity2, entity3
539        for _ in 0..5 {
540            let e1 = Arc::clone(&entity1);
541            handles.push(thread::spawn(move || initialize_item_index(&e1)));
542
543            let e2 = Arc::clone(&entity2);
544            handles.push(thread::spawn(move || initialize_item_index(&e2)));
545
546            let e3 = Arc::clone(&entity3);
547            handles.push(thread::spawn(move || initialize_item_index(&e3)));
548        }
549
550        // Collect all results
551        let results: Vec<_> = handles.into_iter()
552                                     .map(|h| h.join().unwrap())
553                                     .collect();
554
555        // Count occurrences of each index
556        let mut counts = std::collections::HashMap::new();
557        for &result in &results {
558            *counts.entry(result).or_insert(0) += 1;
559        }
560
561        // Should have exactly 3 unique indices
562        assert_eq!(counts.len(), 3, "Should have 3 unique indices");
563
564        // Each index should appear exactly 5 times (one entity, 5 threads)
565        for (&idx, &count) in &counts {
566            assert_eq!(
567                count, 5,
568                "Index {} should appear 5 times, appeared {} times",
569                idx, count
570            );
571        }
572
573        // Global counter should be 3
574        assert_eq!(get_registered_item_count() - initial_registered_items_count, 3);
575
576        // Each entity should have one of the indices
577        let indices: Vec<_> = vec![
578            entity1.load(Ordering::Acquire),
579            entity2.load(Ordering::Acquire),
580            entity3.load(Ordering::Acquire),
581        ];
582
583        let mut sorted_indices = indices.clone();
584        sorted_indices.sort();
585        // As before, we account for the fact that some entities have been
586        // initialized in their `ctors`, so the indices we created don't start at 0.
587        let expected_indices = vec![
588            0 + initial_registered_items_count,
589            1 + initial_registered_items_count,
590            2 + initial_registered_items_count
591        ];
592        assert_eq!(sorted_indices,
593                   expected_indices
594        );
595
596    }
